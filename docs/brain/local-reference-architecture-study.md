# Local Reference Architecture Study

This study is based on the local uploaded archives listed in `local-reference-codebase-inventory.md`. It uses clean-room architectural observations only.

| Pattern | Local source file(s) | Useful for GORKH | Risk | Current GORKH status | Decision |
| --- | --- | --- | --- | --- | --- |
| Closed learning loop with memory, skill creation, conversation recall, and user model | Hermes `README.md`; `agent/memory_manager.py`; `agent/transports/codex_event_projector.py`; `agent/skill_commands.py` | Confirms value of observing traces, reflecting after work, and proposing durable skills/facts | Autonomous memory/skill writes can over-collect sensitive data if not consent-gated | Adaptive Brain observes, reflects after saved sessions, proposes facts/skills, and requires confirmation for sensitive facts/skills | Copy the loop shape, not autonomous application |
| Memory context fencing and streaming scrubber | Hermes `agent/memory_manager.py` | Strong pattern for preventing injected memory scaffolding from leaking to user-visible output | Complex streaming scrubbers can hide data-flow bugs if not tested | GORKH returns structured profile summaries and does not inject raw memory fences | Adopt principle: sanitize recalled context and keep profile summaries structured |
| Skills as filesystem instructions with inline-shell expansion | Hermes `agent/skill_commands.py` | Shows how skills become procedural memory with config injection | Inline shell/config expansion is too dangerous for GORKH v0 | GORKH skills are DB-stored declarative workflow steps only | Explicitly do not copy executable skill behavior |
| Tool loop guardrails and mutating/idempotent tool classes | Hermes `agent/tool_guardrails.py` | Useful taxonomy for safe tool execution and repeated failure detection | Still assumes powerful tools exist | GORKH disables execute/form/browser/private actions and has restricted manifests | Keep restricted tool set; add audit events and manifest validation |
| Approval bridge with deny fallback | Hermes `acp_adapter/permissions.py` | Good pattern: failed/timed-out approval defaults to deny | Approval UX can be bypassed if action surfaces are too broad | GORKH high-risk permissions require approval or are disabled | Keep deny-by-default model |
| Single serialized agent run with lifecycle streams | OpenClaw `docs/concepts/agent-loop.md` | Good for avoiding session races and making runs observable | Long-running tool agents are not a fit for real-time whisper | GORKH live sessions use lifecycle state, generation checks, and late-write suppression | Retain lifecycle/audit, avoid arbitrary tool loops |
| Plugin hooks around prompt/tool/message lifecycle | OpenClaw `docs/concepts/agent-loop.md` | Clean extension-point vocabulary | Untrusted plugin hooks can intercept data or inject actions | GORKH has internal modules only, no plugin runtime | Do not add plugin execution in v0 |
| Tool registry, plugin SDK, manifest separation | OpenClaw `packages/*`; docs grep around plugin/manifest/tool policy | Validates manifest-first tool governance | Malicious skill/plugin installation risk | GORKH has static manifests and restricted executor | Keep static registry; no dynamic plugin install |
| Browser/search separation | OpenClaw docs around browser, web search, sandbox, tools | Supports separating public search/fetch from browser automation | Browser login/cookies/forms are high risk | GORKH has `research` search/fetch and `BROWSER_PROVIDER=none` | Keep web search/fetch only; no browser login/session |
| Full-duplex voice client/server split | PersonaPlex `README.md`; `client/src/protocol/types.ts`; `client/src/pages/Conversation/*`; `moshi/moshi/server.py` | Confirms bridge/control split, protocol messages, audio start/stop lifecycle | Provider/runtime is specialized and model-license-bound; not appropriate inside backend | GORKH uses backend control plane + voice gateway media bridge | Keep provider-agnostic gateway; do not integrate PersonaPlex runtime |
| Persona/role conditioning | PersonaPlex `README.md`; conversation UI model params | Useful conceptually for controlled role/persona prompts | Persona can override safety or manipulate user | GORKH uses safety/playbook/persona boundaries, not free persona role prompts | Only use constrained policy prompts |
| Barge-in/session control | PersonaPlex client socket/audio hooks and protocol control messages | Confirms need for explicit control events and interruption state | Full-duplex audio can imply hidden listening if consent UX is weak | GORKH already has consent-first `/voice`, gateway, and cancel-speech events | Keep explicit consent and barge-in control |

## GORKH-Native Decision

GORKH keeps the useful architecture patterns: auditable observation, saved-session reflection, declarative skill proposals, provider routing, public research/fetch separation, voice gateway separation, and barge-in lifecycle control.

GORKH does not copy or integrate external frameworks, executable skill systems, browser automation, arbitrary shell/code execution, plugin runtimes, PersonaPlex/NVIDIA runtime code, hidden recording, or autonomous high-risk actions.
